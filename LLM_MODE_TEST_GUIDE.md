# 大模型模式测试指南

## 概述

创意小海现在支持两种模式：
- **硬编码模式（默认）**：使用传统的硬编码逻辑
- **大模型模式（新）**：使用大模型智能协调工作流

## 如何切换模式

### 方法1：在API请求中指定

在发送给 `/api/xiaohai/chat` 的请求体中添加 `useLLM` 参数：

```json
{
  "message": "帮我为智能水杯生成一个8秒的视频",
  "useLLM": true
}
```

- `useLLM: true` → 启用大模型模式
- `useLLM: false` → 使用硬编码模式（默认）
- 不传 `useLLM` → 使用环境变量 `DEFAULT_USE_LLM` 的值（默认false）

### 方法2：设置环境变量

在 `.env` 文件中设置：

```
DEFAULT_USE_LLM=true
```

这会设置全局默认模式。

## 测试场景

### 场景1：用户提供产品信息生成脚本

**用户消息**：
```
帮我为智能水杯生成一个8秒的视频
```

**预期结果**：

**硬编码模式**：
- 📋 固定的提示："【识别完成】\n\n正在自动分析参考内容..."
- 🤖 自动执行步骤3、4、5
- 📄 返回3个固定模板脚本

**大模型模式**：
- 🧠 大模型分析用户意图
- 💬 自然的回复："好的！我会基于智能水杯为您创作视频脚本。让我先分析一下，然后生成3个不同风格的脚本供您选择。"
- 📊 返回智能生成的3个脚本（更符合用户需求）

**对比重点**：
- 回复的自然度
- 脚本的相关性和质量
- 响应时间

---

### 场景2：用户反馈脚本太长

**用户消息**：
```
太长了
```

**预期结果**：

**硬编码模式**：
- 🔍 硬编码关键词检测：`['太', '短', '长', '不', '修改', '调整', '换', '重新']`
- ⚡ 快速响应，但可能误判
- 📝 重新生成脚本（简单调整）

**大模型模式**：
- 🧠 深度理解用户意图（"希望缩短时长"）
- 💬 自然的回复："明白了，我来为您生成一个更短的版本，大概8秒左右，让节奏更紧凑一些。"
- 🎯 智能调整脚本（不仅仅是缩短，还优化内容）

**对比重点**：
- 意图理解准确性
- 对话的自然度
- 脚本调整的智能程度

---

### 场景3：用户选择脚本

**用户消息**：
```
第一个不错，就用这个
```

**预期结果**：

**硬编码模式**：
- 🔍 关键词检测："确认"、"好"、"ok"
- 📝 固定确认信息
- 🎬 直接进入视频生成

**大模型模式**：
- 🧠 理解用户的选择
- 💬 自然的确认："好的！您选择了第一个脚本。我现在开始为您生成视频，请稍候..."
- 📊 可能提供额外信息（如预计时长）

**对比重点**：
- 对话的友好度
- 信息提供是否充分

---

### 场景4：用户意外输入

**用户消息**：
```
今天天气不错
```

**预期结果**：

**硬编码模式**：
- ⚠️ 可能无法识别
- 🔄 重复上一步操作或报错
- ❌ 用户体验差

**大模型模式**：
- 🧠 识别这是闲聊
- 💬 友好的回应："是啊，好天气让人心情愉快！不过我们正在创作视频呢，您对刚才的脚本有什么想法吗？或者需要调整吗？"
- 🎯 引导用户回到正题

**对比重点**：
- 异常处理能力
- 用户引导能力

## 回退机制

如果大模型模式出现问题，可以随时回退：

### 方法1：修改前端请求
将 `useLLM` 设置为 `false`

### 方法2：修改环境变量
设置 `DEFAULT_USE_LLM=false`

### 方法3：删除代码
如果需要完全移除大模型模式，可以：
1. 删除 `LLMWorkflowOrchestrator` 相关代码
2. 删除 `DecisionExecutor` 相关代码
3. 恢复原有的硬编码逻辑

## 注意事项

1. **成本**：大模型模式会产生额外费用（LLM调用）
2. **延迟**：大模型模式响应时间可能比硬编码模式长100-500ms
3. **不确定性**：大模型决策可能不稳定，需要容错机制
4. **监控**：密切关注大模型调用的成功率和错误率
