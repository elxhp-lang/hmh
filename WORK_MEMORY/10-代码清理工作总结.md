## 七、经验教训

### 7.1 工作原则

1. **小步快跑**: 分批次清理，每批验证后再进行下一批
2. **验证优先**: 每次修改后立即运行构建和功能测试
3. **备份到位**: 清理前创建 Git 标签，便于回滚
4. **风险评估**: 高风险操作需充分评估后再执行
5. **收益衡量**: 风险 > 收益时果断放弃

### 7.2 常见陷阱

1. **死路径误判**: 工具检测的死路径可能实际是活跃功能
2. **类型错误累积**: 删除文件时注意是否有类型引用
3. **残留引用**: 删除前先全局搜索确认无引用
4. **时机不对**: 系统不稳定期不要进行大型重构

### 7.3 部署环境适配

| 环境类型 | 推荐方案 | 说明 |
|---------|---------|------|
| 单实例 | 内存限流 | 适合传统服务器 |
| 多实例 | Redis 限流 | 需外部存储 |
| Serverless | API 网关限流 | 不适合内存方案 |

### 10.1 拆分 agent-tools-service.ts（方案7）

**原始需求**：文件包含24个工具函数，过于庞大，需要拆分

**详细分析**：

```
文件: src/lib/agent-tools-service.ts
行数: ~1500行
函数数量: 24个
函数分类:
  - 视频生成类 (3): submitVideoTask, queryTaskStatus, batchGenerate
  - 媒体分析类 (4): analyzeVideo, analyzeImage, analyzeMultipleImages, generateFirstFrame
  - 库查询类 (3): getLearningLibrary, getProductReference, searchProduct
  - 用户偏好类 (2): getUserPreference, saveUserPreference
  - 脚本生成类 (2): generateScript, regenerateScript
  - 模板管理类 (3): getTemplates, getTemplate, createTemplate
  - 素材管理类 (4): saveMaterial, deleteMaterial, updateMaterial, getMaterials
  - 协作类 (2): updateCollaborationMemory, getCollaborationStatus
  - 会话类 (2): getSession, clearSession
```

**风险评估**：

| 风险项 | 等级 | 说明 |
|--------|------|------|
| 类型依赖 | 高 | 24个函数涉及大量类型定义，重组后引用关系复杂 |
| 测试覆盖 | 低 | 无单元测试，改动后无法自动验证 |
| 回归测试 | 高 | 需手动测试24个工具的调用链路 |
| 时间成本 | 高 | 预计需要2-3小时重构 + 半天测试 |

**收益评估**：

| 收益项 | 评估 |
|--------|------|
| 可维护性 | 提升约20%（文件更小，但函数关系未变） |
| 加载性能 | 无提升（同模块引用） |
| 团队协作 | 轻微提升（可并行编辑不同子文件） |

**最终结论**：风险 > 收益，暂不执行

**替代方案**：
1. 在文件顶部添加注释分区标记（低风险）
2. 使用 `// #region` 折叠代码块
3. 添加 README.md 说明文件结构
